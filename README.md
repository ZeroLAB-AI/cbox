# cbox

A hardened container that runs OpenAI Codex - and any other AI coding agent -
inside a secure boundary, with one shared on-disk memory that survives session
limits and machine switches.

Codex runs unattended at full speed behind a real security boundary: network-gated,
filesystem-scoped, host untouched. Agents share a single on-disk brain per project,
so work continues across session ends, usage limits, and even across engines. When
you need a second opinion, agents delegate to each other - depth-limited, git-scoped,
fully audited.

## Why cbox

- **Codex, sandboxed and unattended.** Run `codex` with full autonomy inside a
  locked-down container - the container *is* the boundary, so the host stays safe
  even in no-approval, full-access mode.
- **One shared brain and session index across engines.** `LEDGER.md + PROGRESS.md`
  and `.cbox/sessions/` per project persist across session ends and usage limits. Any
  agent reads the same memory on start and continues where the last one stopped -
  no re-briefing.
- **Cross-engine delegation.** Codex delegates to other agents and they delegate
  back, over the shared brain. Every call is depth-limited, restricted to an
  in-scope git worktree, and written to a bytewise audit log.
- **Model routing to cut cost.** Route routine work to cheaper tiers and reserve
  the frontier model for the hard parts. Optional local model (Ollama, Qwen) as a
  zero-cost worker, off by default.
- **Session auto-resume.** A watchdog detects usage limits and resumes the task
  from the shared brain (optional).

## Quick start

```bash
./setup.sh          # one-time wizard, stages all setup
```

Open a new shell (or `source ~/.bashrc`) after setup so the shortcuts load, then,
from a project folder:

```bash
codex                              # start Codex
cbox ai analyse codex              # read-only scan
cbox ai plan codex -p "task"       # plan only
cbox ai full codex -p "task"       # execute the task, unattended
```

The `cbox`, `codex`, `claude`, and `hermes` shortcuts are optional bash functions
generated by setup. If they are not on your PATH - for example if you skipped the
shell-functions step - call the tool by its full path: any `<shortcut>` is just
`/path/to/cbox run <shortcut>` (e.g. `/path/to/cbox run codex`), and a bare
`/path/to/cbox` opens the hub.

`cbox ai` takes a mode (`analyse|plan|full`), an engine (`codex|claude|local-qwen|auto`),
and an optional prompt. All binaries, settings, and policies are pre-configured.
Every Codex call passes an audited guard: an in-scope git worktree, one delegation
hop, and full bytewise logs.

Run a bare `cbox` in a project to open an interactive hub - pick an engine to start
or attach, resume a session, or reach `shell`, `logs`, and `doctor` for the current
project. See [MANUAL.md](MANUAL.md) for details.

## Modes

- **global** (default): one container shared across all workspaces.
- **isolated**: one container per project.

See [MANUAL.md](MANUAL.md) for the setup wizard, storage modes (mount vs volume),
lifecycle, cross-engine delegation, troubleshooting, and per-feature toggles.

## Security

Codex and every other agent run inside the container boundary, so unattended,
full-access execution never touches the host beyond what you explicitly mount.
Guards enforce git-scope on every delegated call, cap delegation depth, and write
bytewise audit logs. Network access is gated, and no secrets are baked into the
image.

## Built with Codex

cbox was built by driving OpenAI Codex on GPT-5.6 through its `codex mcp-server`,
across four delegate tiers picked per task. Most of the implementation, review, and
debugging was delegated to Codex over MCP - the same path the tool exposes at
runtime. See the project's "About" and "How we built it" for details.
