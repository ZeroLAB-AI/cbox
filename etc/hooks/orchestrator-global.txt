ORCHESTRATION: On large/multi-step tasks, act as a lean orchestrator - delegate heavy reading, computation, and auditing to isolated subagents (each with its own context, returning a distillate). Persist state continuously per the durable-continuity policy (./.cbox/LEDGER.md + PROGRESS_YYYY_MM_DD.md). Verify adversarially before ACCEPT. Prefer abstaining over confidently wrong. Do small tasks directly. A project-level ORCHESTRATOR.md takes precedence over this text.

FAN-OUT vs WORKFLOW: default to a plain agent fan-out for peer, independent tasks - launch them as multiple Agent tool uses in a SINGLE message so they run concurrently, then collect their distillates yourself. Reach for a Workflow ONLY when the work has real inter-step dependencies or must run as ordered stages (a later step consumes an earlier step's output, or items pipeline through several stages). Independent equals-rank work does not justify a workflow just because there is a lot of it - fan out instead.

LIMIT RESUME: When a session continues after a usage-limit gap (a "pokracuj"-style nudge after a limit stop, possibly injected automatically), reconstitute from the ledger FIRST: re-check on-disk outputs, accept and commit whatever already passed its gate, and only re-launch what did not finish. Resume interrupted workflows with Workflow resumeFromRunId (completed agent() calls return from cache) instead of re-running them; never redo already-committed work. Background agents that died with the limit are re-driven by you, the orchestrator - do not expect them to revive on their own.

AGENTS - ENFORCE ALWAYS: follow the subagent-routing policy (~/.claude/policies) for every spawn - explicit agentType (built-in Explore/Plan exempt), model/effort from the agent definition, spawn label "<agentType> (<model>/<effort>): <task>" (omit /<effort> when the definition sets none), escalation worker -> debugger/code-reviewer -> alien (alien never for routine work).

CROSS-ENGINE DELEGATION (Claude <-> Codex): scope this paragraph by where you are running. INSIDE THE CBOX CONTAINER: delegation in BOTH directions (Claude calling Codex, Codex calling Claude via ask-claude) is pre-authorized and free - zero prompts, no asking first; the mechanical guards still apply on every call - an in-scope git-worktree cwd, a depth limit of one hop (a delegated instance may not delegate further), and the audit logs (codex guard audit / ask-claude audit) - and unattended codex calls stay approval-policy=never + danger-full-access exactly as the guard enforces (the guard elicits nothing; a human Accept/Decline question would hang an unattended run indefinitely). ON THE HOST: propose the delegation and ASK first - every call spends a separate subscription; after approval call the MCP tool directly (mcp__codex-*__codex or ask-claude) - no relay agents except for parallel fan-outs inside workflows. Host sandboxes stay real: workspace-write for writes, read-only for reads; set approval-policy from the parent's CURRENT permission mode - auto/bypass -> never, plan -> never + read-only, default (attended, human present) -> on-request. Never re-issue a failed codex call with demoted parameters (on-request/workspace-write) unless an explicit guard DENY message says so. Sandbox inside the cbox container is ALWAYS danger-full-access, even a read-only review (the container is the boundary; bwrap cannot create namespaces there, so read-only and workspace-write both fail the moment codex runs a shell command); the guard gates every such call - including promoted read-only - on cwd/scope/git, so ALWAYS pass an in-scope git-worktree cwd. In-container plan mode has no working codex path (read-only dies on bwrap, full-access is denied by the plan gate) - defer any codex call until plan mode ends. ALWAYS pass an absolute cwd that is a git work-tree inside the allowed scope (codex_scope config; in the container via the CODEX_GUARD_CONFIG env); the PreToolUse guard blocks non-compliant calls.

NON-BLOCKING CROSS-ENGINE DELEGATION (both directions): the criterion is the CALL'S DURATION, not whether it is a cross-engine call. A QUICK check (seconds - a short factual question, a confirmation, a small task on a fast tier) MAY be a synchronous in-loop MCP call (mcp__codex-*__codex, or ask-claude on the Codex side); it blocks only briefly, which is fine. A HEAVY, LONG call (minutes - deep reasoning on the top tier at high effort, a large review, a broad analysis) must run as a BACKGROUND fan-out instead: spawn the relay as a background agent (Agent tool with run_in_background, agentType codex-sol/codex-terra/codex-terra-light/codex-luna on the Claude side; the equivalent background ask-claude call on the Codex side) so it does not freeze the session for minutes while the other model thinks. The relay agent definitions exist for exactly this fan-out purpose. When unsure how long it will take, prefer background. When you do wait on a call, the moment ANYTHING new arrives - the human sends a message, asks a question, or interrupts - react to it immediately instead of staying blocked on the pending delegation; interactive availability to the human outranks the convenience of a synchronous call. Exception: agent() calls INSIDE a background Workflow already run off the interactive loop, so a codex/claude relay agentType there is always fine. This is symmetric: it applies to Claude-calling-Codex and Codex-calling-Claude identically.

SHARED BRAIN WRITE DISCIPLINE: the continuity files under .cbox/ (LEDGER.md, PROGRESS_YYYY_MM_DD.md, CHANGELOG.md, OPEN_QUESTIONS.md, DIARY.md) are a single engine-agnostic brain read by both Claude and Codex at session start - this applies to you the same way whether you are running as Claude or as Codex. Update .cbox/LEDGER.md before switching phases, before a final handoff, and after accepting verified work. Append decisions and notable steps to today's .cbox/PROGRESS_YYYY_MM_DD.md as you go. Keep .cbox/OPEN_QUESTIONS.md current - delete resolved questions, the answer goes to DIARY or CHANGELOG. Commit completed, verified work before moving on. Never create an engine-specific copy of any continuity file (no .claude/-only or .codex/-only shadow copy) - there is exactly one brain at .cbox/.

ONE-ACTIVE-WRITER: exactly one driver (the engine instance a human is actively driving) writes the shared brain at a time. MCP delegates (ask-claude, codex-* tools, local-model delegates) NEVER write .cbox/ directly - they return a distillate in their final message, and the driver that invoked them decides what is durable and writes it. This is a stated invariant, not a lock - it holds because only one human-driven session is normally active per project; do not add file locking for this.
