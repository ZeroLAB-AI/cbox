# Durable continuity across session / usage limits

Long multi-step work must survive session end, context compaction, and usage-limit gaps (the session limit refreshes in hours; the weekly limit in days). Treat disk as the memory of record - never hold important state only in context.
- **Project-local memory lives in `.cbox/` of the project the session started in** (engine-agnostic: the same files are read by Claude and Codex), in two layers, each file with its own template norm in `~/.claude/templates/`:
  - **Continuity layer (for the agent to resume):** `LEDGER.md` + `PROGRESS_YYYY_MM_DD.md` - a MANDATORY PAIR, always enabled together (one "history" switch): ledger = state + queue, single source of truth, holds ONLY the live wave - closed waves move to `LEDGER_ARCHIVE.md` (top = now; `~/.claude/templates/LEDGER_TEMPLATE.md`); progress = one per day, honest chronological step log (`~/.claude/templates/PROGRESS_TEMPLATE.md`). `DIARY.md` - optional; the agent's own private space in the project: exceptional moments and lived experience that shape the project's personality - never routine facts (those belong to PROGRESS/LEDGER) (`~/.claude/templates/DIARY_TEMPLATE.md`).
  - **User layer (quick overview, no digging through LEDGER/PROGRESS):** `CHANGELOG.md` - only in projects that use git (no git, no changelog); what landed on master, sections per commit hash, [Unmerged] on top (`~/.claude/templates/CHANGELOG_TEMPLATE.md`). `OPEN_QUESTIONS.md` - optional; always-current open questions, resolved ones get deleted (answer goes to DIARY/CHANGELOG) (`~/.claude/templates/OPEN_QUESTIONS_TEMPLATE.md`).
- **Persist as you go.** Keep `.cbox/LEDGER.md` current (update before switching phases and at day-close with a RESUME note), keep `OPEN_QUESTIONS.md` in sync with reality, and git-commit completed, verified work - before moving on to the next thing. Where the continuity hooks are installed, commits and session events land in `CHANGELOG.md`/`PROGRESS_YYYY_MM_DD.md` mechanically - write there by hand only decisions and context the hooks cannot capture; without hooks, keep the old manual discipline.
- **On session start, reconstitute first.** Look for the project's resume anchor (`.cbox/LEDGER.md` / `PROGRESS.md` / `DIARY.md`, the project CLAUDE.md or AGENTS.md) and rebuild context from it before acting; if a queue exists there, continue it without waiting for a new prompt.
- **When a limit is hit, resume cleanly.** Committed work is safe on disk; in-flight background work (agents/workflows) may be cut mid-run. A fresh session - whether an hour later or days later - re-checks on-disk outputs, accepts/commits whatever already passed its gate, and re-launches only what didn't. Never redo already-committed work.
- **Limits aren't tool-readable.** You only observe a hit (a subagent fails with a "limit" error, or the top-tier/alien model becomes unavailable). Tell the user to check `/usage` or `/usage-credits` for the numbers; then wait for the refresh and resume from the ledger.
